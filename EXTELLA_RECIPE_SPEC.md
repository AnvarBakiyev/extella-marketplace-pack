# Формат рецепта Extella (`.extella.json`)

Единый **декларативный** формат установки/запуска для всего, что есть в витрине:
приложения, CLI, локальные сервисы. В отличие от рецептов Pinokio (это исполняемый
JS → нужен Node + песочница), **рецепт Extella — данные, не код.** Раннер читает JSON
и выполняет заранее заданные шаги. Никакого `eval`, никакого Node на устройстве клиента.

Pinokio-рецепты мы **конвертируем в этот формат офлайн** (через песочный резолвер
`recipe_resolve.js` у нас на сборке) — клиент получает уже безопасный JSON.

## Схема

```json
{
  "id": "sillytavern",
  "name": "SillyTavern",
  "kind": "app",                       // app | cli | service | model
  "source": { "git": "https://github.com/SillyTavern/SillyTavern" },
                                         // git-репо; ИЛИ {"none": true} для чистой установки без клона
  "requires": ["node", "git"],          // рантаймы для авто-бутстрапа: git|node|python|uv|conda
  "install": [                          // шаги установки по порядку
    {
      "run": "npm install",             // shell-команда
      "cwd": "app",                     // папка относительно корня приложения (опц.)
      "venv": "env",                    // python venv для шага (опц.; создаётся автоматически)
      "env": { "FOO": "bar" },          // доп. переменные окружения (опц.)
      "when": { "platform": "darwin" }  // условие: platform|gpu|arch == значение (опц.)
    }
  ],
  "start": {                            // как запускать (для kind app|service)
    "run": "node server",
    "cwd": "app",
    "venv": "env",
    "env": { "SILLY_TAVERN_PORT": "{{port}}" },
    "daemon": true,                     // долгоживущий сервер
    "ready": { "log": "https?://\\S+" } // как понять готовность+порт: регэксп URL в логе (приём Pinokio on:event)
  }
}
```

## Подстановки (в строках `run` и значениях `env`)
- `{{port}}` — свободный порт, выданный раннером (пробрасывается в env приложения)
- `{{gpu}}` — `apple` | `nvidia` | `cpu`
- `{{platform}}` — `darwin` | `linux` | `win32`
- `{{arch}}` — `arm64` | `x64`
- `{{root}}` — абсолютный путь корня установки

## Условия `when`
Шаг выполняется, только если ВСЕ пары совпадают с окружением, напр.
`{"gpu": "nvidia"}` — только на NVIDIA; `{"platform": "darwin"}` — только macOS.

## Детект готовности `ready`
- `{"log": "<regex>"}` — раннер следит за `server.log`; при совпадении берёт **порт из
  найденного URL** и считает сервис поднятым (так приложение само сообщает реальный порт).
- `{"port": 8080}` — фиксированный порт: ждать, пока он не начнёт слушать.
- нет `ready` — для `kind: cli` (установка без запуска).

## kind
- `app` — GUI/веб-приложение (клон + install + start-daemon + открыть iframe)
- `cli` — консольная утилита (только install; проверка `which`)
- `service` — фоновый сервис (как app, но без UI-окна)
- `model` — локальная модель (через существующие cap_localmodel_* / hf_space_install)

## Почему это лучше Pinokio для витрины
1. **Безопасность:** данные, а не код → на клиенте нет исполнения чужого JS, песочница не нужна.
2. **Без Node на клиенте:** раннер на чистом Python; Node ставится только если сам рецепт (`requires`) его просит.
3. **Единый формат:** одним рецептом описываем и приложение, и CLI, и сервис — не только Pinokio-приложения.
4. **Совместимость:** любой Pinokio-рецепт конвертируется в этот формат нашим резолвером.
